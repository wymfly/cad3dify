# V3 技术调研：基础库评估 + 开源项目深度评估

> **日期:** 2026-02-26（更新：深度调研完成）
> **目的:** 评估 cad3dify 现有基础能否支撑 V3 愿景，深入调研可替代/可借鉴的开源项目
> **关联:** V3 愿景设计见 `2026-02-26-v3-vision-text-to-printable.md`
> **结论:** cad3dify 继续作为基础，无可替代项目，但需整合外部数据集和训练方法

---

## 一、cad3dify 现有基础评估

### 1.1 整体复用度：约 75%

| 模块 | 复用度 | 说明 |
|------|-------|------|
| pipeline.py（管道骨架） | 85% | 阶段串联、callback、fallback 机制直接复用 |
| 代码执行器 agents.py | 95% | 通用 Python 执行，V3 直接复用 |
| validators（验证体系） | 90% | 参数校验 + 包围盒 + 几何验证全部复用 |
| DrawingSpec 数据模型 | 80% | Pydantic 继承扩展为 PreciseSpec |
| ModelingStrategist（策略选择） | 85% | Jaccard 匹配算法复用，扩展支持模板 |
| ChatModelParameters | 85% | 多模型提供商支持 |
| 知识库示例 | 70% | 标签体系复用，代码需模板化 |
| 建模策略指南 | 75% | 策略文本复用，代码片段需适配 |
| render.py | 70% | 需增加 STL/3MF 转换 |
| SmartRefiner | 60% | 图片输入场景复用，纯文字场景不需要 |
| V1 管道 | 20% | 仅作 fallback |

### 1.2 CadQuery 耦合度：中等偏高

**运行时硬依赖仅 2 个文件**（render.py、validators.py），可通过 `CADBackend` 接口抽象。

**真正的深度耦合在知识库**：20 个示例代码 + 7 个策略指南全部围绕 CadQuery API。替换 CAD 引擎 = 重写知识库（2-4 周工作量）。

**结论：短期保留 CadQuery，通过接口层预留替换能力。**

### 1.3 关键优势

1. **管道已模块化** — Stage 1 可替换为文字意图解析，Stage 2 可替换为模板引擎
2. **知识库设计良好** — PartType + features 标签 + Jaccard 匹配可直接扩展
3. **验证体系成熟** — 四层防线只需增加可打印性检查
4. **参数化模板演进可行** — 现有示例代码已将数值提取为顶部变量，天然适合模板化

### 1.4 最大技术债

**`features: list[dict]`** — 特征用非结构化 dict 表示，但 `part_types.py` 中已有未使用的结构化模型（`HolePatternSpec`, `FilletSpec`, `ChamferSpec`）。V3 Phase B 开始时应首先统一。

## 二、开源项目深度评估

### 2.1 参数化 CAD 框架

| 框架 | Stars | CAD 内核 | STEP | Python | LLM 训练数据 | 推荐 |
|------|-------|---------|------|--------|-------------|------|
| **CadQuery** | 4,600 | OCCT（工业级） | ✅ | ✅ | ~450K（三个数据集合计） | **首选** |
| **Build123d** | 1,400 | OCCT（工业级） | ✅ | ✅ | 几乎为零 | 不推荐迁移 |
| **OpenSCAD** | 8,916 | CGAL（受限） | ❌ | ❌ DSL | 社区最大 | 仅消费级原型 |

**Build123d vs CadQuery 深入对比：**

Build123d API 设计更 Pythonic（上下文管理器 vs 方法链），类型提示更完善。但在 LLM 生态中处于严重劣势：CadQuery 拥有 Text2CAD 170K + Text-to-CadQuery 170K + CAD-Coder 110K ≈ **450K 标注数据**，而 Build123d 仅有一个社区个人演示项目，数据量可忽略。通用 LLM（GPT-4/Claude）对 CadQuery 的理解远好于 Build123d。

**结论：CadQuery 是当前唯一兼具工业精度和 LLM 生态的选择。**

---

### 2.2 CADAM — 最热门的 text-to-3D 工具

| 属性 | 评估 |
|------|-----|
| Stars | 1,223（5 个月积累） |
| 技术栈 | React 19 + Three.js + OpenSCAD WASM + Supabase Edge Functions |
| LLM | 通过 OpenRouter 支持 Claude/Gemini 等多模型 |
| CAD 引擎 | **OpenSCAD WASM**（浏览器端运行，零服务器成本） |
| 许可证 | **GPL-3.0**（copyleft，衍生作品必须开源） |

**核心优势：**
- **浏览器端完全运行**：WASM 编译，毫秒级参数调整反馈
- **参数滑块 UI**：LLM 生成代码后自动提取变量为参数，滑块实时调整无需重新调用 AI
- 验证了"text → 参数化 3D"的市场需求

**致命短板：**
- **不支持 STEP 输出**：OpenSCAD/CGAL 基于 CSG/mesh，不支持 B-Rep/NURBS
- **精度上限受限**：适合 3D 打印原型，不适合精密零件
- **GPL-3.0 限制商用**：闭源产品集成需开源全部衍生代码

**替换 CAD 引擎可行性：** 难度高，约需重构 40-50% 核心代码。OpenSCAD WASM 的优势在于浏览器端零成本运行，换 CadQuery 意味着必须增加后端编译服务。

**借鉴价值：** 仅 UI/UX 设计思路（参数滑块、实时预览交互），不使用其代码（GPL 约束）。

---

### 2.3 CAD-Coder (MIT, IDETC 2025) — 图像 → CadQuery

| 属性 | 评估 |
|------|-----|
| Stars | 125 |
| 方法 | LLaVA 1.5（CLIP-ViT-L + Vicuna-13B）微调 |
| 数据集 | GenCAD-Code 163K 图像-代码对 |
| 精度 | 100% 语法正确率，IOU_best = 0.675 |
| 许可证 | Apache-2.0（代码），模型权重含 LLaMA 限制 |

**关键发现（深度调研修正初步结论）：**

1. **操作覆盖极其有限**：仅 4 种基本操作（Line/Arc/Circle/Extrude），**不支持** boolean、fillet、chamfer、revolve、sweep、loft。这意味着无法生成工业常用零件。
2. **代码质量非惯用**：矩形草图用 4 条 line 而非 `.rect()`，代码冗长，是 GenCAD 命令的直译。
3. **IOU 0.675 含义**：生成模型与 ground truth 平均 67.5% 体积重叠，对精密零件远不够。
4. **仅图像输入**：不支持文本描述，论文明确将文本输入列为"未来工作"。

**数据集实际可用性：**

| 用途 | 可行性 | 说明 |
|------|-------|------|
| 直接用于 RAG | 中等 | 需补充自然语言描述 + 重构代码风格 + 扩展操作类型 |
| 微调基线 | 高（受限于操作范围） | 163K 规模已验证有效，但覆盖面太窄 |
| 直接部署 | 低 | 13B 模型，需 26GB 显存，无集成方案 |

### 2.4 Text-to-CadQuery — **最值得关注**

| 属性 | 评估 |
|------|-----|
| 论文 | arXiv:2505.06507 |
| 方法 | 在 Text2CAD 170K 数据集上微调 LLM 直接生成 CadQuery 代码 |
| 最优模型 | **Qwen2.5-3B**（全参数 SFT），Median CD = 0.191×10⁻³ |
| 数据集构建成本 | **仅 $170，9 小时**（Gemini 2.0 Flash 转换） |

**为什么最值得关注：**

1. **范式性突破**：从"生成命令序列向量"转为"生成可执行 CadQuery 代码"，代码天然可执行、验证、编辑
2. **极高性价比**：3B 参数超越 7B Mistral，成本效益极高
3. **数据集构建管道可复用**：DeepCAD JSON → Gemini 转 CadQuery → 执行验证 → 视觉评估，全流程自动化
4. **比 Text2CAD baseline 降低 48.6% Chamfer Distance**

**模型对比：**

| 模型 | 参数量 | Median CD (×10³) | Mean CD (×10³) |
|------|--------|------------------|----------------|
| Text2CAD (baseline) | - | 0.370 | 26.417 |
| **Text-to-CadQuery (Qwen2.5-3B)** | 3B | **0.191** | **10.229** |
| CAD-Coder NeurIPS25 (Qwen2.5-7B+RL) | 7B | **0.170** | **6.540** |

**局限：** 仅文本输入，170K 数据集限制天花板，OpenCascade 浮点精度导致约 15% 执行失败。

### 2.5 CAD-Coder (NeurIPS 2025) — 强化学习增强

> ⚠️ 与 2.3 同名但不同团队，arXiv:2505.19713

| 属性 | 评估 |
|------|-----|
| 方法 | Text-to-CadQuery 基础 + SFT + GRPO（几何奖励强化学习） |
| 模型 | Qwen2.5-7B-Instruct |
| 数据 | 110K text-CadQuery 三元组 + 1.5K CoT 样本 |
| 成果 | Mean CD 6.54（比 Text-to-CadQuery 再降 36%），无效率仅 1.45% |

**创新点：**
- Chain-of-Thought 推理使脚本无效率降低 50%+
- 几何奖励函数：CD < 10⁻⁵ 满分，CD > 0.5 零分

**问题：** 代码/权重公开状态不明确。如果开源，是当前精度最高的 text-to-CadQuery 方案。

---

### 2.6 Text2CAD (NeurIPS 2024) — 数据集之母

| 属性 | 评估 |
|------|-----|
| 数据集 | 170K 模型 + 660K 文本标注（三级复杂度） |
| 方法 | Transformer 自回归生成 CAD 命令序列向量 |
| 输出 | **命令序列向量**（不是代码） |

**核心价值：** 定义了 text-to-CAD 基准任务，170K+660K 数据集被后续所有重要工作复用。模型本身已被 Text-to-CadQuery 和 CAD-Coder 大幅超越。

**结论：** 数据集有价值，模型不使用。

### 2.7 CAD-MLLM (NeurIPS 2024) — 多模态但不可用

| 属性 | 评估 |
|------|-----|
| 数据集 | Omni-CAD 450K（文本+多视角图像+点云） |
| 输入 | 文本、图像、点云，或任意组合 |
| 代码发布 | **推理代码未发布、模型权重未发布** |

多模态输入能力是其核心差异化，但当前完全不可用。仅作为架构参考。

### 2.8 PartCAD — 包管理器，非 AI 方案

| 属性 | 评估 |
|------|-----|
| Stars | 433 |
| 方法 | YAML 配置 → 调用通用 LLM → 生成代码 |
| AI 质量控制 | **无**（无验证、无编译检查、无回归测试） |
| 活跃度 | 最后更新 2024-12 |

本质是通用 LLM 的简单封装，不是领域特化方案。包管理理念有价值，但 AI 生成能力不可靠。

## 三、关键结论

### 3.1 无可替代项目

**没有一个开源项目能替代 cad3dify 完成终极目标。** 逐一排除：

| 候选 | 排除原因 |
|------|---------|
| CADAM | OpenSCAD 内核无 STEP，GPL-3.0 限商用 |
| CAD-Coder (MIT) | 仅 4 种操作，仅图像输入，代码非惯用 |
| Text-to-CadQuery | 仅文本输入，无管道/验证/UI，仅学术模型 |
| CAD-Coder (NeurIPS25) | 最优精度但权重未开源 |
| Text2CAD | 输出向量非代码 |
| CAD-MLLM | 推理代码未发布 |
| PartCAD | AI 是 LLM 简单封装，开发停滞 |

### 3.2 cad3dify 的独特价值

cad3dify 是唯一同时具备以下能力的项目：

1. **CadQuery/OCCT 工业级内核** — 与所有学术项目相同的基础
2. **完整管道**（分析 → 策略选择 → 代码生成 → 执行 → 验证 → 修复） — 其他项目都只做"生成"
3. **四层验证体系** — 参数校验 + 包围盒 + VL 比较 + 拓扑验证
4. **知识库 + 特征匹配** — 20 个 Tagged Example + Jaccard 相似度选择
5. **自愈能力**（SmartRefiner） — 生成错误后自动诊断 + 修复循环

学术项目聚焦"生成精度"，cad3dify 聚焦"端到端可靠性"。两者互补。

### 3.3 推荐方向：cad3dify 作为运行时 + 外部 AI 增强

```
┌──────────────────────────────────────────────────┐
│  cad3dify 保留层（运行时 + 质量保障）                  │
│  ├─ pipeline.py（管道编排）                          │
│  ├─ validators（参数/包围盒/几何/拓扑验证）            │
│  ├─ SmartRefiner（VL 比较 + 自愈循环）               │
│  ├─ agents.py（安全代码执行）                        │
│  ├─ render.py（3D 渲染 + STL/3MF 导出）             │
│  └─ 知识库（TaggedExample → ParametricTemplate 演进）│
├──────────────────────────────────────────────────┤
│  外部增强层（新建或整合）                              │
│  ├─ IntentParser（自然语言 → DrawingSpec）           │
│  │   └─ 基于 Qwen2.5-Coder 或通用 LLM              │
│  ├─ 代码生成增强                                     │
│  │   ├─ 路线 A: RAG（Text-to-CadQuery 170K 数据集）  │
│  │   └─ 路线 B: 微调（SFT + GRPO，参考 NeurIPS25）   │
│  ├─ 参数化模板引擎（消除 LLM 代码生成不确定性）         │
│  ├─ 工程标准知识库（螺栓/法兰/公差推荐）               │
│  └─ 参数滑块 UI（借鉴 CADAM 设计，自主实现）           │
└──────────────────────────────────────────────────┘
```

### 3.4 代码生成增强的两条路线对比

| 维度 | 路线 A: RAG | 路线 B: 微调 |
|------|-----------|------------|
| 数据需求 | 170K 作为检索库 | 110K-170K 作为训练集 |
| 启动成本 | 低（向量化 + 检索即可） | 高（GPU 训练，4×H100 约 10 小时） |
| 精度上限 | 受限于检索质量和 LLM 能力 | 更高（Median CD 0.17×10⁻³） |
| 可维护性 | 高（增删检索库即更新） | 中（需重训练） |
| 适合阶段 | V3 Phase B（快速验证） | V3 Phase C/D（精度攻坚） |

**推荐：先 A 后 B。** 路线 A 快速验证可行性（1-2 周），路线 B 在需要突破精度瓶颈时启用。

## 四、行动建议

### 4.1 确认继续 cad3dify 的理由

1. 无可替代 — 没有现成项目同时覆盖"生成+验证+修复"
2. 75% 复用 — 管道、验证、执行器、知识库均可直接使用
3. CadQuery/OCCT 正确 — 所有工业级项目都指向 OCCT 内核
4. 学术项目提供的是"AI 层"能力，cad3dify 提供的是"工程层"能力，两者互补

### 4.2 优先级调整后的下一步

| 优先级 | 行动 | 依据 |
|-------|------|------|
| **P0** | 实现 STEP → STL/3MF 导出 | 满足 3D 打印最基本需求，CadQuery 已有能力只需封装 |
| **P0** | 下载 Text-to-CadQuery 170K 数据集 | 评估数据质量，为 RAG 做准备 |
| **P1** | 统一 features 为结构化模型 | 消除最大技术债，为模板化奠基 |
| **P1** | 实现 3-5 个参数化模板（法兰/阶梯轴/平板） | 验证"模板+填参"精度路线 |
| **P2** | 实现 RAG 检索（170K 数据集 → 向量化 → Few-shot） | 提升 LLM 代码生成质量 |
| **P2** | 原型 IntentParser（文字 → DrawingSpec） | 开通自然语言输入 |
| **P3** | 评估微调路线（SFT + GRPO） | 精度攻坚 |
| **P3** | 参数滑块 UI 原型 | 交互式参数确认 |
